Watir: Web Application Testing in Ruby

What is Watir?

Watir (pronounced water) stands for "Web Application Testing in Ruby." It is a library that enables you to automate tests for Web applications using Internet Explorer. Watir can do anything with IE that's supported by the InternetExplorer object . That is, it can fill in forms, press buttons, follow hyperlinks, etc.

Watir also lets you check results. For example, you can check whether certain text appears on the page, so you can take the appropriate action.

What are the advantages of using it over other tools?

  • It's a Free Open Source tool (there are no costs to use the tool)
  • There's a very active and growing community behind it
  • It's powerful
  • It uses Ruby, a full featured Object Oriented scripting language, rather than a proprietary vendorscript.
  • It's easy to use

What can I do with it?

Anything you can do with Ruby, a keyboard, a mouse, and Internet Explorer.

How do I get started?

  1. Get Ruby (visit rubyinstaller.rubyforge.org/wiki/wiki.pl, then click download)
  2. Get Watir (from the wtr.rubyforge.org page, download the zip file with the "Latest Build"
  3. Look over Watir's API (which can be found in The Watir User Guide), figure out what you want to do to what control on a page, start typing your script, and run it!

What are Watir's system requirements?

You need to be running Windows with IE and ruby (1.8.2).

Can I run all my Watir scripts from a command line?

Sure! You can have an all.rb that runs through all your scripts.

What is the "right" way to use it?

Wrap all your tests in Ruby's Test::Unit framework and assert based on what you expect (and don't expect) there to be a web page (See testunit.talbott.ws for more on Test::Unit)

What can't it do?

Currently, it cannot handle browsers other than IE, or handle file uploads, but work is underway to support these features.

How can I get help on it?

Visit the wtr-general mailing list (which can be found at rubyforge.org/mailman/listinfo/wtr-general)

What are some good practices?

Tests should all start from a known state - that is, open a new browser window / new session. Tests should run independently of each other with a setup and teardown for each test.
